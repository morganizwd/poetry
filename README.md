# Интеллектуальная система генерации стихов

Учебный проект по генерации русскоязычных стихотворений. В проекте реализованы два подхода:

1. Генерация с помощью цепей Маркова.
2. Генерация с помощью LSTM-модели с простым механизмом attention.

Проект рассчитан на запуск в Google Colab. Основная генерация выполняется двумя локальными моделями: цепями Маркова и LSTM. Дополнительно поддерживается опциональный LLM-слой через Gemini API: он не заменяет генераторы, а редактирует уже созданный черновик и оценивает смысловую связность по фиксированной шкале.

## Простое описание для клиента

Эта система учится на готовом наборе стихотворений и затем пробует создавать новые строки в похожем стиле.

В проекте есть два варианта генерации:

- Цепи Маркова быстро собирают новые строки на основе вероятных переходов между словами.
- LSTM обучается предсказывать следующее слово по предыдущим словам и за счёт этого строит строки более гибко.

Дополнительно система пытается подбирать рифмы по окончаниям слов. Пользователь может выбрать схему рифмовки, например `AABB`, `ABAB`, `ABBA` или `AAAA`.

В расширенном варианте Markov и LSTM сначала создают черновики, а затем LLM может выступить как редактор:

- сохранить количество строк и схему рифмовки;
- сохранить исходную тему и первую строку;
- исправить грамматику;
- сделать текст более связным и осмысленным.

После генерации выводятся простые метрики:

- насколько часто удалось подобрать рифму;
- насколько похожи окончания рифмующихся строк;
- сколько уникальных строк получилось;
- средняя длина строки;
- количество гласных;
- количество уникальных слов в стихотворении.

Метрики не являются строгой литературной оценкой качества. Они нужны, чтобы примерно сравнивать результаты разных запусков.

Для оценки осмысленности добавлен отдельный LLM-оценщик. Он получает стихотворение без указания модели-генератора и выставляет оценки от 1 до 5 по критериям: смысловая связность, грамматика, сохранение темы, поэтичность и общее качество.

### 6. Запуск LSTM В Блокноте

Для LSTM в блокноте есть два запуска.

В этой версии LSTM обучается не на перевёрнутых отдельных строках, а на фрагментах стихотворений в обычном порядке слов. Между строками добавляется служебный токен `<line>`. Это помогает модели видеть переходы между строками и не ломает естественный порядок фразы ради рифмы.

Сначала быстрая проверка:

```python
quick_lstm = LSTMRhymingPoetryGenerator(model_name="poet_quick")

quick_lstm.load_and_train(
    "poems_clean.txt",
    epochs=1,
    batch_size=16,
    max_vocab_size=3000,
    max_training_lines=2000,
)
```

Она нужна, чтобы быстро убедиться, что обучение реально идёт. В выводе должны появиться строки:

```text
Обучение модели ...
Батчей в одной эпохе: ...
Epoch 1/1
   batch 1/...
Epoch 1/1, Loss: ...
```

После быстрой проверки можно запускать основной вариант:

```python
full_lstm = LSTMRhymingPoetryGenerator(model_name="poet")

full_lstm.load_and_train(
    "poems_clean.txt",
    epochs=20,
    batch_size=32,
    max_vocab_size=12000,
    max_training_lines=20000,
)
```

При генерации LSTM делает так:

- генерирует строку слева направо;
- для рифмующейся строки создаёт несколько кандидатов;
- выбирает кандидата, у которого последнее слово лучше рифмуется;
- использует `top_k` sampling, чтобы не выбирать совсем случайные слова из всего словаря;
- отбрасывает явно плохие строки: повторы подряд, слишком много служебных слов, строку с предлогом или союзом в конце.

Если нужно быстрее, можно уменьшить `epochs`:

```python
epochs=3
```

Если нужен более долгий финальный запуск, можно увеличить:

```python
epochs=30
```

В основном запуске используется стартовая строка:

```python
START_LINE = "Я вышел ночью в сад пустой"
```

Она не делает текст идеально осмысленным, но задаёт генерации начальное направление. Также используется более спокойная температура:

```python
temperature=0.65
```

Это уменьшает случайность и обычно делает строки чуть связнее. Для учебной демонстрации обычно достаточно `20` эпох. Если Colab работает медленно, можно временно поставить `10`.

### 8. LLM-редактор и оценка осмысленности

После генерации Markov и LSTM блокнот может запустить LLM-слой. Для этого нужен ключ Gemini API.

В Colab удобнее всего добавить ключ в `Secrets`:

```text
GEMINI_API_KEY
```

По умолчанию используется модель:

```python
LLM_MODEL_NAME = "gemini-2.5-flash-lite"
```

LLM используется в двух ролях.

Первая роль — редактор. Он получает черновик и инструкцию:

```text
Сохрани количество строк.
Сохрани схему рифмовки AABB.
Сохрани первую строку.
Исправь грамматику.
Сделай текст более осмысленным.
Не пиши стихотворение с нуля.
```

Вторая роль — оценщик. Он получает только текст стихотворения и первую строку, но не получает название генератора. Оценка выставляется по шкале от 1 до 5:

- `semantic_coherence` — смысловая связность;
- `grammar` — грамматическая правильность;
- `theme_consistency` — сохранение темы первой строки;
- `poetic_quality` — художественность;
- `overall` — общее качество.

Так сравнение делится на два этапа:

1. Markov и LSTM сравниваются до LLM, чтобы увидеть качество базовых генераторов.
2. Markov + LLM и LSTM + LLM сравниваются после редактирования, чтобы увидеть, какой черновик лучше улучшается редактором.

Если ключа нет, этот раздел в блокноте будет пропущен.

### 9. Многократный Эксперимент На 10-20 Генераций

В первой ячейке настроек есть параметры:

```python
EXPERIMENT_RUNS = 10
EXPERIMENT_LLM_RUNS = 10
GENERATION_RETRIES = 50
REPORTS_DIR = "/content/poetry/reports"
AUTO_DOWNLOAD_REPORTS = False
```

`EXPERIMENT_RUNS` задаёт, сколько стихотворений сгенерировать каждой базовой моделью. По умолчанию создаётся 10 черновиков Markov и 10 черновиков LSTM. Для более строгого сравнения можно поставить:

```python
EXPERIMENT_RUNS = 20
```

`GENERATION_RETRIES` нужен потому, что LSTM иногда возвращает меньше строк, чем запрошено. Теперь notebook не берёт первый неполный результат, а повторяет генерацию и старается получить ровно `POEM_LINES` строк.

`EXPERIMENT_LLM_RUNS` задаёт, сколько запусков отправлять в LLM-редактор и LLM-оценщик. Если есть ограничения API или долго выполняется оценка, это число можно уменьшить, например:

```python
EXPERIMENT_LLM_RUNS = 3
```

В результате появляются две дополнительные таблицы:

- средние формальные метрики по серии генераций;
- средние LLM-оценки для Markov, LSTM, Markov + LLM и LSTM + LLM.

`REPORTS_DIR` задаёт папку, в которую notebook сохраняет CSV-отчёты.

`AUTO_DOWNLOAD_REPORTS` управляет прямым скачиванием файлов из Colab:

- `False` — notebook только сохраняет отчёты и показывает путь к ним;
- `True` — ячейка скачивания попытается сразу загрузить оба CSV на компьютер пользователя.

### 10. CSV-Отчёт По Полному Циклу

После демонстрационных и batch-запусков notebook формирует полный CSV-отчёт.

Создаются два файла:

1. `poetry_report_detailed_<timestamp>.csv`
2. `poetry_report_summary_<timestamp>.csv`

Подробный CSV содержит:

- режим запуска: `demo` или `batch`;
- номер запуска;
- базовую модель: Markov или LSTM;
- версию текста: сырой черновик или `+ LLM`;
- стартовую строку и схему рифмовки;
- число запрошенных и реально полученных строк;
- полный текст стихотворения;
- отдельные колонки `line_1`, `line_2`, ...;
- формальные метрики;
- оценки LLM по смыслу, грамматике, теме и поэтичности;
- комментарий LLM.

Summary CSV содержит усреднённые значения по версиям стихов и удобен для таблиц сравнения.

В notebook есть отдельные ячейки:

```text
CSV report export
Download CSV reports
```

Первая создаёт файлы, вторая позволяет скачать их из Colab.

## Что вводить при запуске

Оба скрипта работают примерно одинаково.

Можно оставить начальную строку пустой:

```text
Введите начальную строку:

```

Тогда программа начнёт генерацию сама.

Можно задать свою первую строку:

```text
Введите начальную строку:
> Я помню чудное мгновенье
```

После этого программа попробует продолжить стихотворение и подобрать рифмы к нужным строкам.

Доступные схемы рифмовки:

```text
AABB - первая строка рифмуется со второй, третья с четвёртой
ABAB - первая строка рифмуется с третьей, вторая с четвёртой
ABBA - первая строка рифмуется с четвёртой, вторая с третьей
AAAA - все строки пытаются рифмоваться между собой
```

## Методика Сравнения И Оценивания

Сравнение проводится в два этапа.

Сначала оцениваются исходные черновики:

```text
Markov
LSTM
```

Этот этап нужен, чтобы честно сравнить сами генераторы до вмешательства LLM.

Затем оцениваются отредактированные версии:

```text
Markov + LLM
LSTM + LLM
```

Этот этап показывает, насколько хорошо черновики разных моделей улучшаются редактором.

Для одного демонстрационного примера используются переменные:

```python
START_LINE = "Я вышел ночью в сад пустой"
RHYME_SCHEME = "AABB"
POEM_LINES = 8
```

Для итогового сравнения используется серия запусков:

```python
EXPERIMENT_RUNS = 10
```

То есть notebook генерирует 10 стихотворений Markov и 10 стихотворений LSTM. Если нужно больше данных для таблицы, значение можно увеличить до 20.

Формальные метрики проверяют структуру текста:

- успешность рифмовки;
- качество рифмы по окончаниям слов;
- уникальность строк;
- среднюю длину строки;
- количество уникальных слов.

Смысловые качества оцениваются отдельно через LLM-оценщик по фиксированной шкале от 1 до 5. Оценщик не получает название генератора, поэтому сравнение остаётся слепым относительно источника текста.

Критерии LLM-оценки:

- `semantic_coherence` — связность смысла;
- `grammar` — грамматическая правильность;
- `theme_consistency` — сохранение темы первой строки;
- `poetic_quality` — художественность;
- `overall` — общее качество.

LLM-оценка не считается абсолютно объективной литературной экспертизой. Это автоматизированная приближённая оценка, удобная для сравнения большого числа результатов. Для дипломной или отчётной работы можно дополнительно вручную проверить несколько случайных примеров.

## Почему LLM Не Используется Как Основной Генератор

Если генерировать стихотворение сразу через GPT-подобную модель, качество текста обычно будет выше, но исследовательская часть проекта исчезнет: система превратится в один вызов внешнего API.

В этой работе важно показать:

- как работает вероятностный подход на цепях Маркова;
- как работает нейросетевой подход на LSTM;
- какие ограничения есть у каждой модели;
- как можно улучшить результат гибридной архитектурой.

Поэтому Markov и LSTM остаются основными генераторами. LLM применяется только после них, как редактор черновика и инструмент оценки осмысленности.

## Краткое сравнение подходов

Цепи Маркова:

- работают быстро;
- проще устроены;
- не требуют долгого обучения;
- иногда дают более механические строки.

LSTM:

- обучается дольше;
- требует TensorFlow;
- лучше учитывает последовательность слов;
- результат сильнее зависит от качества и размера датасета.

LLM-редактор:

- не создаёт стихотворение с нуля;
- улучшает уже созданный черновик;
- повышает грамматичность и смысловую связность;
- помогает получить более качественный финальный текст для продуктовой версии.

В сравнении важно разделять этапы:

- до LLM сравниваются Markov и LSTM как базовые генераторы;
- после LLM сравниваются Markov + LLM и LSTM + LLM как гибридные варианты;
- отдельно оценивается, насколько LLM улучшила каждый исходный черновик.
